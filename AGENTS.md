# AGENTS.md

## What this repo is

A Docker Compose stack (PostgreSQL 16 + pgAdmin + Metabase) for learning/testing,
not production. All lifecycle, config, and backup logic lives in bash scripts at
the repo root — agents should drive the stack through those scripts, not raw
`docker compose` commands (except read-only ones like `logs`/`ps`).

For the full operational runbook (setup, start/stop/restart, extra users,
logins, backups), see `.claude/skills/metastack-ops/SKILL.md` — it's the
source of truth for *how to operate* the stack. This file covers conventions
for *working on the code*.

## Testing

There is no CI and no automated test suite. `test/pg-backup.sh` and
`test/mb-backup.sh` are manual smoke tests for the backup/restore scripts:
they require a running stack (`./setup.sh` first), mutate live data
(`backup_test` table / Metabase site name), and overwrite `test1`/`test2`
backup files. Only run them against a disposable stack, never one holding
data anyone cares about.

## Script conventions

- `set -euo pipefail` and `cd "$(dirname "${BASH_SOURCE[0]}")"` at the top of
  every root-level script, so it can be invoked from any working directory.
- `# SPDX-License-Identifier: AGPL-3.0-only` as the first line, matching
  `LICENSE`.
- Naming: `<component>-<verb>.sh` (`pg-backup.sh`, `mb-restore.sh`, ...).
  Keep this prefix consistent — `pg-` for Postgres, `mb-` for Metabase.
- Comments explain *why*, not *what* — e.g. why a step retries, why a file
  is regenerated every run vs. only when missing. Don't add comments that
  just restate the next line.
- Destructive operations (anything that stops a container and wipes a
  volume, e.g. `*-restore.sh`) print what they're about to do before doing
  it; keep that pattern for new destructive scripts.

## Directories that are not source

- `pg-config/` and `pgadmin-config/`: regenerated by `setup.sh`, wiped by
  `teardown.sh`. Edit `postgres-config.template/` / `pgadmin-config.template/`
  instead if the change should persist.
- `backups-pg/` / `backups-mb/`: backup archives, gitignored except `.keep`.
- `.env` and `extra-users.conf`: gitignored, contain secrets (passwords).
  Never print their full contents in a commit message, PR description, or
  code comment.

## Commits

No enforced format observed in history — plain, descriptive messages
(imperative mood, e.g. "Add Metabase backup/restore scripts").
